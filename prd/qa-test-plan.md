# QA Test Plan: 파일 단위 변경 취소 및 다른 커밋으로 이동

## 목적

아코디언 안 파일 목록에서 우클릭으로 실행하는 두 가지 액션이 사용자 기대대로 동작하는지 확인한다.

- 파일 변경사항 취소
- 다른 커밋으로 옮기기

이번 변경은 Undo 배너, 빈 커밋 유지, 파일 목록 자동 갱신에도 영향을 주므로 그 부분까지 함께 확인한다.

## 권장 준비 상태

아래 조건을 만족하는 jj 저장소 하나를 준비하면 대부분의 시나리오를 빠르게 확인할 수 있다.

1. 변경된 파일이 2개 이상 있는 커밋 1개
2. 변경된 파일이 1개뿐인 커밋 1개
3. 이동 대상이 될 다른 커밋 1개
4. 상태가 각각 `A`, `M`, `D` 인 파일 예시
5. 가능하면 source 커밋의 자식 커밋도 1개 이상

## 핵심 체크리스트

### 1. 파일 행 우클릭 진입

1. 아무 커밋이나 펼쳐서 파일 목록을 본다.
2. 파일 행을 우클릭한다.

기대 결과:

- 파일 행에만 컨텍스트 메뉴가 열린다.
- 메뉴에는 `파일 변경사항 취소`, `다른 커밋으로 옮기기`가 보인다.
- 파일 오른쪽에 항상 보이는 `X` 버튼이나 `...` 버튼은 없다.

### 2. 수정 파일(M) 취소

1. `M` 상태 파일이 있는 커밋을 펼친다.
2. 파일을 우클릭하고 `파일 변경사항 취소`를 누른다.

기대 결과:

- 확인 모달 없이 바로 실행된다.
- 상단에 성공 배너와 `Undo` 버튼이 보인다.
- 펼쳐진 파일 목록이 다시 로드되어 해당 파일이 사라지거나 상태가 바뀐다.
- 커밋 그래프가 새 상태를 반영한다.

### 3. 추가 파일(A) 취소

1. `A` 상태 파일이 있는 커밋에서 같은 액션을 실행한다.

기대 결과:

- 별도 예외 없이 같은 메뉴로 처리된다.
- 해당 파일이 현재 커밋에서 빠진다.
- Undo로 원래 상태를 되돌릴 수 있다.

### 4. 삭제 파일(D) 취소

1. `D` 상태 파일이 있는 커밋에서 같은 액션을 실행한다.

기대 결과:

- 같은 메뉴로 처리된다.
- 현재 커밋 기준으로 삭제가 취소되어 파일이 다시 살아난다.
- Undo가 정상 동작한다.

### 5. 파일 하나만 다른 커밋으로 이동

1. 파일이 여러 개 있는 커밋을 펼친다.
2. 파일 하나를 우클릭하고 `다른 커밋으로 옮기기`를 누른다.
3. 그래프에서 대상 커밋을 클릭한다.
4. 확인 배너에서 `Move`를 누른다.

기대 결과:

- 별도의 파일 선택 모달 없이 바로 destination 선택 흐름으로 들어간다.
- 확인 배너의 파일 수가 `1 files` 로 보인다.
- 이동 완료 후 Undo 배너가 보인다.
- source 커밋 파일 목록에서는 그 파일만 빠지고, 대상 커밋 쪽에는 그 파일 변경이 반영된다.

### 6. 마지막 파일 이동 후 빈 커밋 유지

1. 변경된 파일이 1개뿐인 커밋에서 `다른 커밋으로 옮기기`를 실행한다.

기대 결과:

- source 커밋이 삭제되지 않는다.
- source 커밋은 빈 커밋으로 남는다.
- description, 위치, change id 흐름이 자연스럽게 유지된다.

### 7. 마지막 파일 취소 후 빈 커밋 유지

1. 변경된 파일이 1개뿐인 커밋에서 `파일 변경사항 취소`를 실행한다.

기대 결과:

- source 커밋이 삭제되지 않는다.
- 커밋은 빈 커밋으로 남는다.
- 자식 커밋이 있다면 그 내용은 깨지지 않는다.

### 8. Undo 검증

1. `파일 변경사항 취소` 실행 직후 `Undo`를 누른다.
2. `다른 커밋으로 옮기기` 실행 직후 `Undo`를 누른다.

기대 결과:

- 두 경우 모두 원래 그래프 상태로 돌아간다.
- 파일 목록도 다시 원상복구된다.
- Undo 뒤에 stale한 파일 목록이 남지 않는다.

### 9. immutable 커밋 보호

1. immutable 커밋을 펼친다.
2. 파일 행을 우클릭한다.

기대 결과:

- 파일 액션 메뉴는 보여도 항목이 비활성화되어 있거나, 실행되지 않는다.
- rewrite가 실제로 발생하지 않는다.

### 10. 진행 중 상태 잠금

1. 파일 이동 destination 선택 중이거나, rebase 중인 상태를 만든다.
2. 다른 커밋을 펼치거나 파일 행 우클릭을 시도한다.

기대 결과:

- 추가 파일 액션을 시작할 수 없다.
- 이미 펼쳐져 있던 파일 목록이 조작 중 UI와 충돌하지 않는다.

### 11. 자동 갱신

1. 파일 액션을 성공시킨 뒤 아코디언을 닫지 않고 그대로 본다.

기대 결과:

- 파일 목록이 새로고침되어 최신 상태를 보여준다.
- 새로 고침을 위해 페이지를 다시 열 필요가 없다.

### 12. 에러 표시

1. 의도적으로 실패하는 상황을 만든다.
2. 파일 액션을 실행한다.

기대 결과:

- 기존 에러 배너를 통해 실패 이유가 표시된다.
- UI가 destination 선택 상태나 실행 중 상태에 갇히지 않는다.

## 회귀 체크

이번 작업에서 파일 액션 배너를 건드렸기 때문에 아래도 빠르게 확인한다.

1. 기존 `Split` 실행 후 성공 배너와 `Undo`가 보이는지
2. 기존 `Squash into parent` 실행 후 성공 배너와 `Undo`가 보이는지
3. 기존 `Move changes from here` 실행 흐름이 여전히 동작하는지
4. Describe 저장 후 그래프가 새로고침되는지

## 확인 메모

테스트하면서 아래 관찰값을 남기면 좋다.

1. 어떤 상태의 파일(`A`, `M`, `D`)에서 확인했는지
2. source 커밋이 단일 파일 커밋이었는지
3. source 커밋에 자식 커밋이 있었는지
4. Undo가 정상 복구됐는지
5. 그래프와 파일 목록 중 어느 쪽에서 이상이 보였는지
